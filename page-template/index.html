Costco Wholesale — Gasoline Sales Forecasting Report

The data science & analytics team at Costco had an existing report tracking gasoline sales volumes at all locations worldwide across different grades of fuel. However, the report lacked helpful visualizations, making it hard to understand and communicate user needs effectively.

Abstract

During my internship at Costco, I was tasked with leading a project that aimed at improving the gasoline sales/forecasting dashboard, which lacked effective data visualizations and useful analysis tooling.

Over 10 weeks, I worked with the data science team to create a dashboard that is not only easy to navigate, but works as a tool that can help support purchasing decisions and can generate performance summaries. I conducted thorough UX audits, interviews, and affinity diagrams to pinpoint user needs and pain points, worked within the constraints of Power BI, and iteratively crafted low, mid, and high fidelity wireframes that incorporated frequent feedback from stakeholders.

The final designs successfully addressed key issues and introduced new features such as a designated page for location specific comparisons and analysis. Although the engineering team had to rework the designs in order to test in QA, this project initiated conversations on the potential benefits of a design guideline refresh to enhance the accessibility, usability, and functionality of future data reports.

Introduction

The abandoned data dashboard

Hosted on Power BI, the gasoline sales/forecasting dashboard was no longer being used and did not run current data. Although we don't know for which reasons, we can safely assume that visually, the dashboard lacked good design and clear usability.

The purchasing team was manually computing data for further analysis

On a weekly basis, the data science team would send the gasoline purchasing team a CSV file that contained upper and lower forecasting threshold values for specific locations and fuel types. Using this file, they then would then make separate calculations for their analysis. This was a issue because:

  • Having too many csv files clutters real fast and can make it hard to reference older data reports
  • There are no visual indications of performance, only numerical values
  • It takes a lot of time sort through the data and create new functions

Research

Finding user experience and interface design gaps

Not knowing the goals and function for each page yet, I wanted to perform a ux audit based only on the ui and usability. By doing this, I was able to note current pain points, needs, and baseline questions for interview scripting.

Interviews: intended usage and behaviors

By speaking with 3 important individuals — the product owner, lead data scientist, and gasoline central director — each with distinct roles, I gained a comprehensive understanding of the product and current behaviors from varied user perspectives. Some questions I asked were:

  • Could you walk me through how you intended to use the forecasting report?
  • Are there specific metrics or data points you frequently refer to in the forecasting report? If so, how do you use them?
  • Using the forecasting data, what specific tasks or objectives are you typically trying to accomplish?

Affinity diagram insights

I used the affinity diagram as a way to summarize my findings and draw connections from the interviews. This highlighted main problems and helped with which solutions should be prioritized.

User stories: who is this for?

Writing out user stories helps us stay focused on the people who will use the dashboards. It makes sure that what we're designing meets their needs, makes their work easier, and leaves them with a good product experience.

Project definition

An intuitive data analysis tool with features that help make informed decisions while also serving as a resource for high-level performance overviews.

Understanding dashboards

Learning from successful data dashboards

To gain a solid understanding in designing for data, I wanted to reference products that already have successful dashboards such as Google Analytics and Shopify. By studying these examples, I can pick up on common design decisions and best practices, giving me a good baseline for understanding how to craft effective data dashboards.

Power BI limits customizations, sort of

There aren't too many customizable options on power bi and some have very complex workarounds that would be difficult to replicate for future dashboards. For example, to enable custom fonts, you would need to update the JSON file and for users to even see it, the font would have to already be installed on their systems.

To work with these constraints, I educated myself further on Power BI's capabilities through Dashboard in a Day, planned for simple and clean designs, and tested accessibility on native fonts and color palettes.

Articles that helped me understand data and dashboards better:

  • Medium: 10 rules for better dashboard design
  • Harvard.edu: Data visualizations, charts, and graphs
  • Datawrapper: Which fonts to use for your charts and tables

De-cluttering the site map

We're dealing with large data and numerical values, so it can get overwhelming quick. By condensing, simplifying, and reorganizing the site map, we're making sure navigation is intuitive even to someone who may be accessing the report for the first time.

Outlining

Exploring plausible features

Based on the insights from the affinity diagram and the extensive possibilities for data analysis, I prioritized and categorized the features to be added. This process clarified the essential elements to include when creating the drafts.

Dashboard skeletons

Taking inspiration from successful dashboards and by wanting to incorporate new features, I wanted to test out different formats before jumping straight to mid fidelity wireframes. This step helped me map out each sections' placement, helping me design easier and with purpose.

Iterations

Finalizing features through mid fidelity wireframes

Feeling confident with the structural maps, I moved on to drafting the mid fidelity wireframes. With these wireframes, I focused not only on varying placements, but also explored different data visualizations suited to their respective data sets.

Design review #1 feedback:

  • Due to scope, KPIs and region map views were no longer viable
  • Make graph components more accessible, include legends and line variety
  • Prioritize features that can be implemented quickly without needing to create new SQL functions

Minimum viable products with data visualization explorations

I approached these high fidelity wireframes as minimum viable products, only keeping elements that are necessary and can be shipped relatively quickly. I did some color explorations, but we ultimately landed on using native Power BI color palettes.

Design review #2 feedback:

In order to start development and remain on track, these designs were close to final. The second design review mainly involved minor adjustments such as scraping the box and whiskers graph, refining line graph behaviors, and adding another slicer.

Proposed designs

Approved dashboard pages and features

After some refinements, a last minute feature addition, and a final walkthrough with my manager, we landed on these final designs. Below you'll find a dashboard with data analysis tools that help with purchasing decisions while also acting as a resource for high-level summaries.

Forecasting summary page:

The forecast summary page showcases a high level overview of gasoline sales and forecasts in comparison to last year's numbers.

Planning page:

The planning page is a more granular display of sales performance by location, fuel type, and/or date range.

Analysis page:

The analysis page is specifically for comparing data between multiple locations. An example use case would be to see how a new gasoline station is performing compared to others within a 5 mile radius.

In development

Different design, same usability

In order for the dashboard to go through QA, the engineering team had to redesign them in accordance to Costco's design guidelines. Aside from the evident design alterations, some substitutions, and the fixed 16:9 aspect ratio pages, the usability and features of each page remains consistent with the initial proposals.

Recommendations

Design review process

When I joined this team, the ux design sector was relatively new and there wasn't a set design process in place. My approach to design reviews were asynchronous, letting people add their feedback whenever they can. However looking back, having a dedicated meeting time would've been more productive, facilitating quicker feedback and room for open discussions.

Learnings and impact

‍Working with design constraints is very feasible

I was fortunate enough that in my past experiences, I had full creative freedom in my projects. This was the first time being limited to design to software capabilities, and honestly, it wasn't as bad as I thought it was going to be. I may have had to omit certain designs, but I learned about data accessibility and tabular spacing. This project helped improve my ability to adapt, compromise, and educate myself on things I'm unfamiliar with.
